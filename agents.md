# Codex Notes For PicFlow

## Что это за проект

PicFlow - локальная Python-программа для подготовки фотографий и картинок перед публикацией в паблике. Основные задачи:

- поиск и удаление дублей;
- отбор понравившихся и отбраковка остальных;
- категоризация одобренных изображений;
- экспорт по папкам категорий.

Проект работает с локальной библиотекой файлов, хранит индекс в SQLite и даёт простой веб-интерфейс для ручного ревью.

## Стек и точки входа

- Python 3.12+
- SQLite
- Pillow, numpy, requests
- локальный HTTP-сервер без внешнего фреймворка

Главная CLI-точка входа: `python -m picflow ...`

Основные команды:

- `python -m picflow runserver`
- `python -m picflow scan`
- `python -m picflow candidates`
- `python -m picflow verify --limit 200`
- `python -m picflow categorize-ai --limit 100`
- `python -m picflow export-plan`
- `python -m picflow export-apply`

## Ключевые модули

- `picflow/config.py` - конфиг, структура директорий, `.env`
- `picflow/db.py` - SQLite-схема и все запросы
- `picflow/hashing.py` - хеши, превью, подготовка изображений
- `picflow/duplicates.py` - сканирование, поиск дублей, план и применение действий
- `picflow/selection.py` - good/bad отбор входящих файлов
- `picflow/categorization.py` - AI и ручная категоризация approved файлов, экспорт
- `picflow/web.py` - HTTP API и раздача UI
- `picflow/static/*`, `picflow/templates/*` - фронтенд интерфейса

## Рабочий поток приложения

1. `scan` индексирует библиотеку и обновляет `images`.
2. `candidates` строит кандидатов в дубли в `duplicate_candidates`.
3. `verify` проставляет AI-метки для пар дублей.
4. UI `/duplicates` позволяет вручную подтвердить или отклонить дубли.
5. UI `/selection` размечает входящие изображения как good/bad и переносит их в:
   - `approved_unsorted`
   - `rejected_pool`
6. UI `/categorize` или `categorize-ai` ставит категорию approved-файлам.
7. `export-apply` переносит размеченные approved-файлы в `export/<category>`.

## Инварианты, которые нельзя ломать

- `library_root` - источник истины для файловой библиотеки.
- Роли файлов определяются по верхнему уровню пути: `all_photos`, `export`, `approved_unsorted`, `rejected_pool`, остальные папки считаются входящими.
- Для дублей приоритет сохранения выше у `reference`, затем `export`, затем `approved`, затем `incoming`, затем `rejected`.
- `duplicate_action` по умолчанию `delete`, поэтому любые правки вокруг удаления нужно делать особенно осторожно.
- `scan_library()` должен корректно переиспользовать старые записи по `mtime_ns` и `size_bytes`.
- Любые операции перемещения/удаления должны оставлять БД и файловую систему в согласованном состоянии.
- В selection без явной метки файл считается `bad`.
- В categorization файлы с `blocked` не должны уходить в экспорт.

## Практика правок

- Сначала изучать затронутый поток целиком: CLI -> DB -> файловые действия -> API -> UI.
- Если меняется схема или семантика записей `images` / `duplicate_candidates`, обязательно проверить существующие тесты и совместимость с уже созданной SQLite-базой.
- Если меняется логика планирования действий, отдельно проверять коллизии имён файлов.
- Если меняется UI, проверять связанный JS и соответствующий `/api/...` endpoint вместе, а не по отдельности.
- Не редактировать пользовательские секреты в `.env` и реальные рабочие пути без явной причины.
- Учитывать, что в репозитории может быть грязное дерево; не перетирать несвязанные пользовательские изменения.

## Что проверять после изменений

Минимум:

- `pytest`

Если менялась логика дублей:

- `pytest tests/test_duplicates.py tests/test_hashing.py tests/test_smoke_unittest.py`

Если менялись selection/categorization/web:

- `pytest tests/test_smoke_unittest.py`
- ручная проверка через `python -m picflow runserver`

Ручные сценарии для веба:

- открывается dashboard;
- работает список дублей;
- можно поставить manual label;
- selection корректно переносит good в `approved_unsorted`, bad в `rejected_pool`;
- categorization не экспортирует `blocked`;
- export создаёт файлы в ожидаемых папках и не ломается на коллизиях имён.

## Что важно помнить Codex

- Это проект про файловые операции над реальной библиотекой изображений, поэтому цена ошибки выше обычного рефакторинга.
- Предпочитать маленькие, проверяемые изменения.
- При неясности в поведении сначала смотреть тесты: они хорошо отражают бизнес-правила проекта.
- Если задача касается дублей, отбора или сортировки, почти всегда нужно учитывать одновременно БД, путь файла и роль директории.
